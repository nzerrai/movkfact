# S2.2 Type Detection Algorithm - Complete Clarification

**Goal:** Everyone on the team understands EXACTLY how type detection works

---

## 🎯 Algorithm Overview in Plain English

```
For EACH column in CSV:
  1. Read the column header (e.g., "email_address")
  2. Read up to 100 sample values (skip nulls)
  3. For EACH detector (13 total: Email, FirstName, Amount, etc.):
     a. Score: How well does the header match this detector's patterns?
     b. Score: How many sample values match this detector's pattern?
     c. Combine scores: confidence = 0.4 * header_score + 0.6 * value_score
  4. Rank all 13 detectors by confidence score
  5. Return:
     - Primary type: Highest confidence detector
     - Alternatives: Top 2 other detectors (if confidence > 20%)
     - Rationale: WHY we think it's this type
```

## 🎯 Confidence Ranges & User Actions

### **Confidence Levels**

```
Confidence Range | Status | User Action | Shown as
-----------------|--------|-------------|----------
80-100%          | HIGH   | Accept & use | Primary suggestion (green ✓)
60-79%           | MEDIUM | Review      | Primary + show 1-2 alternatives (yellow ⚠)
50-59%           | LOW    | Likely wrong| Show top 3 alternatives (orange ⚠⚠)
0-49%            | UNKNOWN| Must choose | "Unknown type - please select" (red ✗)
```

### **Decision Rules**

```python
def determine_user_action(confidence: int, column_name: str):
    if confidence >= 80:
        return "ACCEPT"  # Show as default, user can override
    elif confidence >= 60:
        return "REVIEW"  # Show primary + alternatives
    elif confidence >= 50:
        return "SELECT"  # Show top 3, user must pick
    else:
        return "UNKNOWN" # User types in the column type
```

## 🧮 Header Scoring Logic (Detailed)

### **Example 1: Email Header Matching**

```
Column Header Input: "email_address"
EmailTypeDetector patterns: ["email", "mail", "e-mail", "contact_email"]

Step 1: Normalize header
  "email_address" → lowercase → "email_address"
                 → remove underscores → "emailaddress"

Step 2: Try exact match
  "emailaddress" == "email" ? NO
  "emailaddress" == "mail" ? NO
  "emailaddress" == "e-mail" ? NO
  "emailaddress" == "emailaddress" (from "contact_email") ? MAYBE

Step 3: Try contains match
  Does "emailaddress" CONTAIN "email" ? YES ✓
  
  How much does it contain?
  - "email" is at start? YES → +20 points
  - "email" is exact prefix? YES → +30 points (exact!)
  
  Score: 20 + 30 + 48 (base) = 98/100
```

## 🔄 Combined Scoring Formula

### **The Math**

```
Formula: Combined_Confidence = (Header_Score × 0.4) + (Value_Score × 0.6)

Why these weights?
  - Headers are explicit but often ambiguous ("name", "date")
  - Values are more reliable indicators of actual data type
  - Users typically put SOMETHING in headers, but values are ground truth
  - Ratio: 40% header, 60% values
```

### **Practical Examples**

**Scenario A: Strong signal from both header and values**
```
Header Score: 95 (header "email" matches perfectly)
Value Score: 98 (almost all values are emails)

Combined: 0.4 × 95 + 0.6 × 98 = 38 + 58.8 = 96.8% ← VERY CONFIDENT
```

**Scenario B: Header says one thing, values say another**
```
Header: "person_name" (strongly suggests FULL_NAME)
  → Header Score for EmailDetector: 0%

Values: All are emails
  → Value Score for EmailDetector: 95%

Combined: 0.4 × 0 + 0.6 × 95 = 57% ← UNCERTAIN
(System says: "Header suggests name, but values look like emails")
```

**Scenario C: Generic header, clear values**
```
Header: "data" (too generic, matches nothing well)
  → Header Score: 5%

Values: All are phone numbers
  → Value Score: 92%

Combined: 0.4 × 5 + 0.6 × 92 = 2 + 55.2 = 57.2% ← UNCERTAIN
(System says: "Can't tell from header, but values look like phone numbers")
```

## 🔍 Running All 13 Detectors in Parallel

### **Pseudocode**

```
function detectColumn(columnHeader, sampleValues):
  detectors = [
    EmailTypeDetector(),
    FirstNameTypeDetector(),
    LastNameTypeDetector(),
    PhoneTypeDetector(),
    GenderTypeDetector(),
    AddressTypeDetector(),
    AccountNumberTypeDetector(),
    AmountTypeDetector(),
    CurrencyTypeDetector(),
    DateBirthTypeDetector(),
    DateCreatedTypeDetector(),
    TimezoneTypeDetector(),
    TimeTypeDetector()
  ]
  
  // Run all 13 detectors in PARALLEL (not sequentially!)
  results = detectors.parallelStream()
    .map(detector → detector.detect(columnHeader, sampleValues))
    .collect(Collectors.toList())
  
  // Sort by confidence (highest first)
  results.sort((r1, r2) → compare(r2.confidence, r1.confidence))
  
  // Return: primary + top 2 alternatives
  primary = results.get(0)
  alternatives = results.stream()
    .skip(1)
    .filter(r → r.confidence > 20)  // Only show if non-trivial
    .limit(2)
    .collect(Collectors.toList())
  
  return DetectionResult(primary, alternatives)
```

## ⚡ Performance Optimization: Sampling Strategy

### **Problem: Full File Scanning is Slow**

```
CSV file: 10,000 rows × 30 columns

Naive approach: Read ALL 10,000 rows for analysis
  - Time: ~500ms just to read + parse
  - Memory: Entire file in memory

Better approach: Sample representative rows
```

### **Sampling Strategy**

```
Total rows in file: N

If N <= 100:
  Sample all rows (file is small anyway)
  
If N > 100:
  Sample:
    - First 50 rows (capture header patterns, early anomalies)
    - Every 10th row up to row 1000 (N=95 more rows)
    - Random 50 rows from middle/end (capture distribution)
    
  Total sampled rows: ~195 (vs 10,000 original)
  Representative? YES (covers start, middle, end)
  Time: ~50ms (10x faster)
```

### **Code Example**

```java
List<String> sampleValues(List<String> allValues, int sampleSize) {
  if (allValues.size() <= sampleSize) {
    return allValues;  // Use all
  }
  
  List<String> samples = new ArrayList<>();
  
  // First 50
  samples.addAll(allValues.subList(0, Math.min(50, allValues.size())));
  
  // Every 10th
  for (int i = 100; i < Math.min(1000, allValues.size()); i += 10) {
    samples.add(allValues.get(i));
  }
  
  // Random from end
  Random rand = new Random();
  for (int i = 0; i < 50 && i < allValues.size(); i++) {
    int idx = rand.nextInt(allValues.size());
    samples.add(allValues.get(idx));
  }
  
  return samples;
}
```

## 🛡️ Edge Case Handling

### **Case 1: Column with All Nulls**

```
Column: "unknown"
Values: [null, null, "", null]

All detectors:
  - Header: Some may match (detector-specific)
  - Values: 0 matches (no non-null values)
  - Combined: Low confidence (mostly header-dependent)

Result: Return UNKNOWN if all detectors < 50%
```

### **Case 2: Mixed Encoding**

```
Column values: ["Francois", "François", "Müller", "Jose", "José"]

Mostly ASCII: 3/5 = 60%
UTF-8 encoded: 2/5 = 40%

Detector treats all as FIRST_NAME if pattern matches?
→ Should handle mixed encoding gracefully
→ Auto-detect file encoding first, then analyze values
```

### **Case 3: Header Only, No Data**

```
CSV content:
"first_name","email","amount"
(no rows below header)

Sample values: [] (empty)

All detectors:
  - Header scores may be high (consistent headers)
  - Value scores: 0 (no values to analyze)
  
Result: Return UNKNOWN for all columns
("Can't tell from header alone, need data values")
```

## 📋 Testing Algorithm Accuracy

### **Measurement Method**

```
Test CSV file with ground truth labels

Example test file:
Header: first_name, email, amount, date_birth
Ground truth: FIRST_NAME, EMAIL, AMOUNT, DATE_BIRTH

Run detection algorithm:
Detected: FIRST_NAME, EMAIL, AMOUNT, DATE_BIRTH

Accuracy = columns_correct / total_columns = 4/4 = 100%
```

### **Partial Credit for Alternatives**

```
If detection returns UNCERTAIN but correct type in alternatives:

Column: "date" (ground truth: DATE_CREATED)
Detected: DATE_BIRTH (62%)
Alternatives: [DATE_CREATED (61%), DATE_EVENT (50%)]

Count as:
- Primary correct? NO (wrong type)
- In alternatives? YES ✓ (partial credit 0.5 points)

Scoring:
Full accuracy = (primary_correct + 0.5*partial_correct) / total
             = (3 + 0.5*1) / 4
             = 3.5 / 4
             = 87.5% accuracy ✓
```
