## Ce qui a bien marché ✅

### 1. Architecture contraintes extensible
La décision d'utiliser `Map<String, Object> constraints` dans `ColumnConfigDTO` est particulièrement réussie. Aucune migration Flyway nécessaire, extensible à tout nouveau type de générateur. Pattern réutilisable pour les prochains épics.

### 2. Drag-and-drop @dnd-kit
Intégration de `@dnd-kit/sortable` dans le wizard étape 2 fonctionnelle et robuste. IDs stables via format `${timestamp}-${random36}` garantissant l'unicité pour le réordonnancement.

### 3. Séparation Preview / Génération standard
`DataPreviewService` sans `@Transactional`, sans dépendance repository — propre architecturalement. Réutilise `GeneratorFactory` existant sans duplication.

### 4. Code review adversariale efficace
14 issues identifiées dont 8/9 HIGH+MEDIUM corrigées avant merge. La review a apporté de la valeur réelle, notamment sur la sécurité (H1 SecurityConfig) et la robustesse des tests (H4, M1).

### 5. Couverture de tests sérieuse
19 tests backend (10 unitaires DataPreviewService + 9 RestAssured) et 21 Jest wizard. Les notes de dev détaillées dans les story specs ont facilité la rédaction des tests.

### 6. Choix qualité > scope
La réduction volontaire du scope (Epic 7 only, 13/30 pts) pour livrer avec qualité maximale était le bon trade-off. Le wizard est stable et production-ready.

## Challenges et zones de croissance 🔴

### 1. Chaîne de 3 causes sur le 403 /api/datasets/preview
Le bug post-review le plus complexe de l'épic a eu **trois causes indépendantes** :
1. SecurityConfig trop permissive (`/api/datasets/**`) → H1 fix avec `HttpMethod.POST` bloquait le preflight CORS OPTIONS
2. JVM chargait l'ancienne SecurityConfig (recompilée sans redémarrage)
3. Frontend envoyait `rowCount`/`dataSetName` au lieu de `numberOfRows`/`datasetName`

Chaque cause isolée était triviale. Combinées, elles ont créé un blocage opaque difficile à diagnostiquer.

### 2. Mismatch de contrat API frontend↔backend
`rowCount` vs `numberOfRows`, `dataSetName` vs `datasetName` — détecté uniquement à la code review manuelle post-implémentation. Pas d'outillage pour vérifier automatiquement le contrat.

### 3. Limitations jsdom non documentées
`crypto.randomUUID()` absent en environnement Node.js/jsdom — découvert par trial-and-error pendant les tests S7.2. Remplacé par `String(Date.now() + Math.random())` puis amélioré en `${Date.now()}-${Math.random().toString(36).slice(2,9)}`.

### 4. npm peer dependency conflicts
`@dnd-kit` sur CRA nécessite `--legacy-peer-deps` — limitation non documentée dans les dev notes initiales.

### 5. 6 failures Jest pré-existantes non résolues
`BatchJobsContext.test.js` et `ConfigurationPanel.test.jsx` — confirmées non-régressives (pré-existantes via `git stash`), mais elles polluent le signal qualité de la suite. Suite à 315/321 au lieu de 321/321.

## Leçons clés

1. **La SecurityConfig est un point de friction récurrent** — l'intégrer explicitement dans le template de story comme étape obligatoire (pas optionnelle) pour tout nouvel endpoint
2. **Les mismatches de contrat API sont systémiques** — pas un problème individuel, un problème d'outillage et de discipline de vérification
3. **Scope réduit + qualité maximale = bon trade-off** — 43% de vélocité sprint mais 100% de story completion avec 476 tests ; le wizard est solide
4. **La code review adversariale a de la valeur** — 14 issues dont 8 critiques corrigées avant merge ; à maintenir pour Epic 8+
